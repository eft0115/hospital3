# 02-ux-flow.md — 스마트허브병원

## 문서 역할

이 문서는 `01-project-summary.md`에서 정의한 스마트허브병원의 브랜드 철학인 **편의성, 문제해결력, 안정감**을 실제 사용자 경험으로 어떻게 체화할 것인지 정의한다.

이 문서는 구현체의 단순 목록이 아니라, 구현이 충족해야 할 **경험의 요구 사양**이다. 따라서 페이지, 컴포넌트, 데이터 모델은 모두 다음 질문에 답해야 한다.

> 사용자가 더 편하게 이해하고, 자신의 문제를 더 빨리 해결하며, 검진 전후 과정에서 더 안정감을 느끼도록 만드는가?

스마트허브병원 웹사이트의 UX는 일반 병원 홈페이지가 아니라 **기업과 근로자를 위한 산업·현장 건강검진 허브**로 설계한다.

---

# 0. UX 설계 전제

## 핵심 사용자 분리

스마트허브병원의 주요 사용자는 크게 세 그룹이다.

1. **기업/산업체 담당자**  
   출장검진, 특수건강검진, 사업장 검진, 협약 문의, 일정 조율이 목적이다.

2. **개인 근로자**  
   배치전검진, 배치후검진, 특수검진 대상 여부, 당일검진, 결과 확인, 준비물 확인이 목적이다.

3. **지역 주민**  
   내과, 정형외과, 신경외과, 재활치료, 예방접종, 건강검진 등 지역 외래 진료가 목적이다.

## UX 핵심 원칙

이 사이트는 모든 사용자를 한 흐름에 넣지 않는다.  
처음부터 사용자의 목적을 분리하고, 각 목적에 맞는 정보와 CTA를 제공한다.

```txt
기업 담당자 → 출장검진 / 특수검진 / 협약 문의
개인 근로자 → 배치전검진 / 당일검진 / 결과 확인 / 준비물
지역 주민 → 외래 진료 / 건강검진 / 검사 안내
```

# 1. 유저 시나리오

각 시나리오는 “사용자가 무엇을 하는가”보다 **사용자가 어떻게 느끼는가**를 우선 기술한다.  
브랜드 가치인 **편의성, 문제해결력, 안정감**이 시나리오 안에서 어떻게 작동하는지도 함께 정의한다.

---

## 시나리오 1. 첫인상

### 상황

사용자가 검색, 블로그, 지도, 광고, 지인 공유 등을 통해 처음 스마트허브병원 웹사이트에 진입한다.

### 주요 사용자

- 기업 출장검진을 알아보는 산업체 담당자
- 배치전검진 병원을 찾는 개인 근로자
- 정왕동/시화/배곧 인근 병원을 찾는 지역 주민

### 사용자가 기대하는 것

사용자는 처음부터 병원의 모든 정보를 보고 싶어 하지 않는다.  
자신의 상황에 맞는 길이 즉시 보이기를 기대한다.

```txt
“내가 여기서 검진을 받을 수 있나?”
“출장검진도 가능한가?”
“배치전검진은 오늘 가능한가?”
“내가 찾던 병원이 맞나?”
```

### 경험의 진행

1. 첫 화면에서 병원의 정체성을 짧게 인식한다.
2. 바로 두 개의 핵심 동선이 보인다.
   - 기업 출장·특수검진 문의
   - 개인 배치전검진 예약
3. 그 아래에 건강검진센터, 진료과, 검사·장비 등 보조 동선이 정리되어 있다.
4. 사용자는 자신이 어디로 가야 하는지 망설이지 않는다.

### 성공의 느낌

```txt
“아, 여기는 검진을 잘 안내해주는 병원이구나.”
“기업 담당자와 개인 검진자가 따로 안내되어 있어서 편하다.”
“내 상황에 맞는 버튼이 바로 보인다.”
```

### 녹아있는 가치

| 가치 | UX 체화 방식 |
|---|---|
| 편의성 | 첫 화면에서 목적별 CTA를 분리한다. |
| 문제해결력 | 사용자의 질문을 서비스명보다 먼저 다룬다. |
| 안정감 | 과장된 의료 광고 대신 명확한 안내와 절차를 보여준다. |

### 필요한 UX 패턴

- 목적 분리형 Hero
- 기업/개인 이중 CTA
- 빠른 서비스 탐색 카드
- 모바일 고정 CTA
- 진료·검진 목적별 진입 경로

---

## 시나리오 2. 시선

### 상황

사용자가 첫 화면을 본 뒤, 어디를 읽어야 하는지 시선을 이동한다.  
병원 웹사이트에서 사용자의 시선은 대부분 오래 머물지 않기 때문에, 중요한 정보가 계층적으로 보여야 한다.

### 주요 사용자

- 검색으로 들어온 개인 근로자
- 검진 가능 여부만 빠르게 확인하려는 기업 담당자
- 병원의 신뢰도를 확인하려는 지역 주민

### 사용자가 기대하는 것

```txt
“복잡하게 읽지 않아도 핵심이 보였으면 좋겠다.”
“내가 받을 검진이 있는지 빠르게 확인하고 싶다.”
“전화하기 전에 준비물이나 절차를 알고 싶다.”
```

### 경험의 진행

1. 사용자는 Hero에서 핵심 문장을 확인한다.
2. 바로 아래에서 자신의 목적에 가까운 카드나 버튼을 찾는다.
3. 스크롤을 내리면 검진 프로세스가 단계별로 보인다.
4. 더 내려가면 자주 묻는 질문과 준비물 안내가 보인다.
5. 사용자는 전화나 문의 전에 불안을 줄인다.

### 성공의 느낌

```txt
“복잡하지 않다.”
“내가 필요한 정보가 어디 있는지 보인다.”
“전화를 해도 뭘 물어봐야 할지 알겠다.”
```

### 녹아있는 가치

| 가치 | UX 체화 방식 |
|---|---|
| 편의성 | 정보 우선순위를 사용자의 목적 순서에 맞춘다. |
| 문제해결력 | 절차, 준비물, 대상 여부, 결과 시점을 한 화면 안에서 연결한다. |
| 안정감 | 사용자가 모르는 상태에서 문의하지 않도록 사전 안내를 제공한다. |

### 필요한 UX 패턴

- 시선 유도형 카드 그리드
- 진행 절차 타임라인
- 요약 정보 박스
- FAQ 아코디언
- 준비물 체크리스트
- 관련 서비스 추천 블록

---

## 시나리오 3. 집중된 Checkout, 목적형 전환

### 상황

사용자는 명확한 목적을 가지고 들어왔다.  
이 문서에서 Checkout은 결제 화면이 아니라 **문의·예약 전환 플로우**를 의미한다.

### 주요 사용자

- 기업 출장검진을 문의하려는 담당자
- 특수건강검진 협약 또는 일정 조율을 원하는 담당자
- 배치전검진을 바로 예약하려는 개인 근로자

### 사용자가 기대하는 것

```txt
“지금 바로 문의할 수 있나?”
“어떤 정보를 입력해야 하나?”
“내가 잘못 신청하는 건 아닌가?”
“신청 후 병원에서 어떻게 연락이 오나?”
```

### 경험의 진행

#### 기업 담당자 흐름

```txt
기업 출장·특수검진 문의 선택
→ 기업/기관 정보 입력
→ 검진 인원 및 희망 일정 입력
→ 검진 종류 선택
→ 담당자 연락처 입력
→ 문의 완료
→ 이후 진행 안내 확인
```

#### 개인 근로자 흐름

```txt
개인 배치전검진 예약 선택
→ 검진 목적 선택
→ 방문 희망일 선택
→ 준비물/금식 안내 확인
→ 연락처 입력
→ 예약 또는 상담 요청 완료
→ 결과 확인 안내 확인
```

### 성공의 느낌

```txt
“내가 맞는 경로로 신청했다.”
“무엇을 준비해야 하는지 알겠다.”
“병원에서 어떤 방식으로 연락이 올지 예상된다.”
```

### 녹아있는 가치

| 가치 | UX 체화 방식 |
|---|---|
| 편의성 | 입력 항목을 사용자 유형별로 최소화한다. |
| 문제해결력 | 신청 전에 필요한 조건과 준비사항을 알려준다. |
| 안정감 | 완료 후 다음 단계와 예상 연락 방식을 명확히 보여준다. |

### 필요한 UX 패턴

- 단계형 문의 폼
- 사용자 유형 선택
- 입력 중 준비물 안내
- 제출 전 요약 확인
- 완료 후 다음 단계 안내
- 전화/카카오/폼 병렬 CTA

---

## 시나리오 4. 목적 없이 사이트 자체를 체험하는 방문자

### 상황

사용자는 당장 예약할 목적이 없다.  
블로그 글을 보다가 공식 웹사이트로 넘어왔거나, 병원이 어떤 곳인지 탐색하는 중이다.

### 주요 사용자

- 지역 주민
- 검진 대상이 될 수 있는 근로자
- 기업 담당자이지만 아직 비교 단계인 사용자
- 병원의 신뢰도와 규모를 확인하려는 사용자

### 사용자가 기대하는 것

```txt
“여기가 어떤 병원인지 알고 싶다.”
“검진을 전문적으로 하는 곳인지 확인하고 싶다.”
“나중에 필요할 때 다시 찾아볼 수 있으면 좋겠다.”
```

### 경험의 진행

1. 사용자는 핵심 서비스와 병원의 방향성을 본다.
2. 건강정보 콘텐츠를 탐색한다.
3. 배치전검진, 특수검진, 출장검진 등 반복되는 주제를 통해 병원의 전문성을 인식한다.
4. 문의를 하지 않아도 병원에 대한 기억이 남는다.
5. 추후 검색이나 필요 시 다시 방문한다.

### 성공의 느낌

```txt
“여기는 검진 안내를 체계적으로 하는 병원이다.”
“나중에 배치전검진이나 기업검진이 필요하면 여기로 보면 되겠다.”
“정보가 정리되어 있어서 믿음이 간다.”
```

### 녹아있는 가치

| 가치 | UX 체화 방식 |
|---|---|
| 편의성 | 콘텐츠를 카테고리별로 탐색하기 쉽게 제공한다. |
| 문제해결력 | 콘텐츠가 단순 글이 아니라 관련 서비스와 연결된다. |
| 안정감 | 정보의 일관성과 의료적 설명 톤으로 신뢰를 만든다. |

### 필요한 UX 패턴

- 건강정보 허브
- 콘텐츠 카테고리 필터
- 관련 서비스 연결 카드
- 관련 FAQ
- 읽기 완료 후 부드러운 CTA
- 북마크 가능한 가이드형 페이지

# 2. 핵심 UX 패턴: 컨셉-패턴 매핑

브랜드의 세 가치가 실제 인터페이스에서 어떻게 구체화되는지 정의한다.  
이 매핑은 어떤 컴포넌트 유형이 필요한지를 유도하는 근거가 된다.

## 2-1. 가치별 UX 요구 사양

| 브랜드 가치 | 경험 요구 사양 | UX 패턴 | 필요한 컴포넌트 유형 |
|---|---|---|---|
| 편의성 | 사용자가 자신의 목적을 빠르게 찾는다. | 목적 분리 진입 | Dual CTA, 목적 선택 카드, 빠른 링크 |
| 편의성 | 검진 전 필요한 정보를 한 번에 확인한다. | 체크리스트 안내 | 준비물 카드, 금식 안내, 결과 시점 박스 |
| 편의성 | 모바일에서 바로 문의할 수 있다. | 지속 노출 CTA | 하단 고정 버튼, 전화/카카오/폼 CTA |
| 문제해결력 | 사용자의 질문을 서비스 구조로 번역한다. | 상황 기반 탐색 | “나는 무엇이 필요한가?” 선택 모듈 |
| 문제해결력 | 복잡한 검진 절차를 단계화한다. | 프로세스 타임라인 | B2B/B2C 프로세스 컴포넌트 |
| 문제해결력 | 콘텐츠가 예약/문의로 이어진다. | 콘텐츠-서비스 브릿지 | 관련 서비스 카드, 관련 FAQ, CTA Band |
| 안정감 | 과장 없이 명확한 안내를 제공한다. | 신뢰형 정보 구조 | 안내문, 범위 명시, 조건 명시, 유의사항 |
| 안정감 | 사용자가 다음 단계를 예측할 수 있다. | 사후 안내 플로우 | 완료 페이지, 연락 예정 안내, 후속 절차 카드 |
| 안정감 | 기업 담당자가 운영 가능성을 판단한다. | 운영 신뢰 제시 | 대응 산업군 카드, 검진 범위, 진행 절차, 문의 폼 |

---

## 2-2. 핵심 UX 패턴 목록

### Pattern 1. Dual Path Entry

기업 담당자와 개인 근로자를 첫 화면에서 분리한다.

```txt
기업 출장·특수검진 문의
개인 배치전검진 예약
```

목적은 사용자가 처음부터 자신의 경로를 찾게 하는 것이다.

---

### Pattern 2. Situation-based Navigation

서비스명을 모르는 사용자를 위해 상황 기반 문구를 제공한다.

예시:

```txt
입사 전 검진이 필요합니다
사업장 출장검진을 알아보고 있습니다
야간근로자 특수검진 대상입니다
건강검진 결과가 필요합니다
근골격계 통증으로 진료를 받고 싶습니다
```

---

### Pattern 3. Process-first Explanation

검진은 결과보다 절차가 불안을 만든다.  
따라서 모든 주요 페이지는 절차를 먼저 보여준다.

```txt
대상 확인
→ 준비물 확인
→ 예약/문의
→ 검진 진행
→ 결과 확인
→ 필요 시 사후관리
```

---

### Pattern 4. Preparation-first UX

개인 근로자에게 가장 중요한 것은 “검진 전에 뭘 준비해야 하는가”이다.

필요 요소:

```txt
신분증
사업장 관련 서류
금식 여부
검진 시간
결과 수령 방식
유효기간
```

---

### Pattern 5. Operational Trust UX

기업 담당자는 의료 설명보다 운영 가능성을 먼저 본다.

필요 요소:

```txt
출장 가능 여부
검진 인원
검진 항목
검진 장소 조건
일정 조율 방식
결과 전달 방식
담당자 연락 방식
```

---

### Pattern 6. Content-to-Action Bridge

블로그/건강정보 콘텐츠는 단순 읽을거리가 아니라 서비스 전환의 진입점이어야 한다.

예시:

```txt
배치전검진 글
→ 배치전검진 안내 페이지
→ 준비물 확인
→ 개인 예약 CTA
```

```txt
특수건강검진 글
→ 특수검진 대상 안내
→ 기업 문의 CTA
```

---

### Pattern 7. Anxiety-reducing FAQ

FAQ는 보조 정보가 아니라 불안을 줄이는 주요 UX 컴포넌트다.

우선 제공해야 할 질문:

```txt
배치전검진은 개인도 받을 수 있나요?
당일검진이 가능한가요?
검진 결과는 언제 나오나요?
금식이 필요한가요?
출장검진은 몇 명부터 가능한가요?
특수건강검진 대상자는 어떻게 확인하나요?
야간근로자 검진은 어떻게 신청하나요?
```

# 3. UX Flow

## 3-1. 전체 경험 플로우

전체 흐름은 “검색 → 이해 → 분기 → 신뢰 → 문의/예약 → 사후 안내”로 구성한다.

```txt
유입
→ 첫인상 인식
→ 사용자 유형 선택
→ 검진/진료 목적 확인
→ 서비스 이해
→ 절차/준비물 확인
→ 신뢰 정보 확인
→ 문의/예약
→ 완료 후 다음 단계 안내
```

각 단계의 감정 목표는 아래와 같다.

| 단계 | 사용자 감정 | UX 목표 |
|---|---|---|
| 유입 | 급함, 불확실함 | 내가 찾던 병원인지 즉시 확인하게 한다. |
| 첫인상 | 탐색, 의심 | 산업·현장 건강검진 허브라는 정체성을 보여준다. |
| 유형 선택 | 혼란 | 기업/개인/지역주민 경로를 분리한다. |
| 목적 확인 | 확인 욕구 | 검진 종류와 대상 여부를 쉽게 확인하게 한다. |
| 서비스 이해 | 불안 감소 | 절차, 준비물, 결과 시점을 보여준다. |
| 신뢰 확인 | 판단 | 운영 능력과 의료 안정성을 확인시킨다. |
| 문의/예약 | 실행 | 최소 입력으로 전환하게 한다. |
| 완료 | 안도 | 다음 단계와 연락 방식을 안내한다. |

---

## 3-2. 기업 담당자 플로우

```txt
검색/블로그/직접유입
→ Home Hero의 [기업 출장·특수검진 문의]
→ 출장검진 또는 특수건강검진 페이지
→ 대상 사업장/검진 항목 확인
→ 진행 절차 확인
→ 가능 산업군 및 대응 범위 확인
→ 문의 폼 작성
→ 완료 후 담당자 연락 예정 안내
```

### 기업 담당자의 핵심 질문

```txt
우리 사업장도 가능한가?
몇 명부터 가능한가?
언제 가능한가?
어떤 항목을 검사할 수 있나?
결과는 어떻게 전달되는가?
누구와 일정 조율을 하나?
```

### 기업 담당자용 전환 컴포넌트

- 기업 출장검진 문의 CTA
- 검진 인원 입력
- 희망 일정 입력
- 검진 유형 선택
- 사업장 위치 입력
- 담당자 연락처 입력
- 개인정보 및 상담 동의
- 제출 후 다음 단계 안내

---

## 3-3. 개인 근로자 플로우

```txt
검색/블로그/지도 유입
→ 배치전·배치후검진 페이지
→ 검진 목적 선택
→ 당일검진/준비물/금식/결과 확인
→ 방문 예약 또는 전화 문의
→ 완료 후 방문 전 체크리스트 확인
```

### 개인 근로자의 핵심 질문

```txt
오늘 받을 수 있나?
준비물이 있나?
금식해야 하나?
결과는 언제 나오나?
회사가 요구한 검진이 맞나?
비용은 어떻게 되나?
```

### 개인 근로자용 전환 컴포넌트

- 개인 배치전검진 예약 CTA
- 준비물 체크리스트
- 금식 안내 카드
- 결과 확인 안내
- 당일검진 가능 여부 안내
- 모바일 전화 CTA
- 카카오 상담 CTA

---

## 3-4. 지역 주민 플로우

```txt
검색/지도/블로그 유입
→ 진료과 또는 건강검진센터 페이지
→ 증상/검사/진료과 확인
→ 병원 위치와 진료시간 확인
→ 전화 또는 방문 문의
```

### 지역 주민의 핵심 질문

```txt
가까운 병원인가?
이 증상도 볼 수 있나?
검사 장비가 있나?
진료시간은 어떻게 되나?
바로 문의할 수 있나?
```

### 지역 주민용 전환 컴포넌트

- 진료과 카드
- 증상 기반 안내 카드
- 진료시간 카드
- 위치 지도
- 전화 CTA
- 검사·장비 연결 카드

# 4. 시간 블렌딩의 전역 흐름

스마트허브병원 UX에서 시간은 매우 중요하다.  
검진 사용자는 대부분 “언제까지 해야 하는가”, “오늘 가능한가”, “결과는 언제 나오는가”를 궁금해한다.

따라서 사이트는 과거, 현재, 미래의 정보를 하나의 흐름으로 연결해야 한다.

## 4-1. 시간 흐름 구조

```txt
과거: 블로그/건강정보/기존 사례
현재: 사용자의 검진 필요 상태
미래: 예약, 방문, 결과 확인, 사후관리
```

## 4-2. 시간 단계별 UX

| 시간 단계 | 사용자 질문 | 필요한 UX |
|---|---|---|
| 검진 전 | 내가 대상자인가? | 대상 확인 카드 |
| 검진 전 | 무엇을 준비해야 하나? | 준비물 체크리스트 |
| 검진 전 | 금식해야 하나? | 금식 안내 카드 |
| 검진 당일 | 어디로 가야 하나? | 위치/진료시간/방문 안내 |
| 검진 후 | 결과는 언제 나오나? | 결과 시점 안내 |
| 검진 후 | 재검이나 사후관리는? | 후속 안내 카드 |
| 기업 문의 후 | 담당자 연락은 어떻게 오나? | 완료 후 프로세스 안내 |

## 4-3. 시간 블렌딩 UX 원칙

1. **검진 전 불안을 줄인다.**  
   준비물, 금식, 비용, 결과 시점을 사전에 안내한다.

2. **검진 중 흐름을 예측 가능하게 만든다.**  
   방문, 접수, 검사, 결과 확인 흐름을 단계화한다.

3. **검진 후에도 경험이 끊기지 않는다.**  
   결과 확인, 재검, 사후관리, 기업 리포트 안내까지 연결한다.

4. **블로그 콘텐츠를 현재 행동으로 연결한다.**  
   과거에 작성된 콘텐츠가 현재의 문의/예약 CTA로 연결되어야 한다.

# 5. 정보구조 IA — 경험 단위 관점

정보구조는 메뉴 목록이 아니라 사용자의 경험 단위로 설계한다.

## 5-1. 경험 단위

| 경험 단위 | 사용자의 질문 | 대표 페이지/섹션 |
|---|---|---|
| 목적 선택 | 나는 어디로 가야 하지? | Home Hero, 목적 선택 카드 |
| 검진 이해 | 어떤 검진이 필요한가? | 배치전·특수·출장검진 페이지 |
| 절차 확인 | 어떻게 진행되는가? | 프로세스 타임라인 |
| 준비 확인 | 무엇을 준비해야 하는가? | 체크리스트, FAQ |
| 신뢰 확인 | 믿을 수 있는가? | 운영 범위, 검사·장비, 병원 안내 |
| 전환 | 어떻게 문의/예약하는가? | 문의·예약 페이지, CTA |
| 사후 안내 | 이후에는 어떻게 되는가? | 완료 페이지, 결과 안내 |
| 콘텐츠 탐색 | 더 알고 싶다 | 건강정보 허브, 콘텐츠 상세 |

## 5-2. IA 구조

```txt

│

│  ├─ 야간근로자 검진
```

# 6. 데이터 모델

데이터 모델은 “어떤 데이터를 저장할 것인가”가 아니라, **어떤 경험을 가능하게 하기 위한 상태인가** 관점으로 정의한다.

---

## 6-1. User Context

사용자의 현재 목적과 상태를 파악하기 위한 모델이다.

| 필드 | 예시 | 경험 목적 |
|---|---|---|
| userType | 기업 담당자 / 개인 근로자 / 지역 주민 | 동선 분리 |
| intent | 출장검진 문의 / 배치전검진 예약 / 진료 문의 | CTA 최적화 |
| urgency | 오늘 / 이번 주 / 일정 조율 필요 | 안내 우선순위 결정 |
| location | 사업장 위치 / 사용자 거주 지역 | 출장 가능성, 지역 정보 제공 |
| serviceNeed | 배치전검진 / 특수검진 / 국가검진 / 외래 | 관련 페이지 추천 |

---

## 6-2. Service Model

각 서비스가 어떤 사용자 경험을 지원하는지 정의한다.

| 필드 | 예시 | 경험 목적 |
|---|---|---|
| serviceName | 배치전검진 | 서비스 인식 |
| serviceCategory | 산업검진 / 외래 / 일반검진 | IA 분류 |
| targetUser | 기업 / 개인 / 지역주민 | 동선 분리 |
| availableMode | 병원 방문 / 출장검진 | 공간 선택 |
| requiredItems | 신분증, 사업장 서류 등 | 준비물 안내 |
| fastingRequired | 예/아니오/상황별 | 검진 전 불안 감소 |
| resultTiming | 당일/며칠 소요/문의 필요 | 결과 기대 관리 |
| primaryCTA | 예약 / 기업 문의 / 전화 | 전환 유도 |

---

## 6-3. Content Article Model

블로그 및 건강정보 콘텐츠를 서비스 전환과 연결하기 위한 모델이다.

| 필드 | 예시 | 경험 목적 |
|---|---|---|
| title | 배치전검진 준비물 안내 | 콘텐츠 인식 |
| category | 배치전검진 가이드 | 콘텐츠 허브 구성 |
| relatedService | 배치전·배치후검진 | 서비스 연결 |
| targetUser | 개인 근로자 | 관련 CTA 선택 |
| keyQuestion | 무엇을 준비해야 하나? | FAQ 연결 |
| recommendedCTA | 개인 배치전검진 예약 | 콘텐츠 후속 행동 |
| relatedFAQ | 금식, 결과, 비용 | 불안 감소 |

---

## 6-4. Inquiry Request Model

문의/예약 플로우를 안정적으로 만들기 위한 모델이다.

| 필드 | 기업 문의 | 개인 예약 | 경험 목적 |
|---|---|---|---|
| requestType | 출장검진 / 특수검진 | 배치전검진 / 외래 | 문의 유형 분리 |
| name | 담당자명 | 신청자명 | 연락 식별 |
| organization | 회사명/기관명 | 선택사항 | B2B 구분 |
| headcount | 예상 인원 | 해당 없음 | 기업 문의 판단 |
| preferredDate | 희망 일정 | 방문 희망일 | 일정 조율 |
| location | 사업장 위치 | 거주/방문 지역 | 출장 가능성 판단 |
| contact | 연락처 | 연락처 | 후속 연락 |
| message | 요청사항 | 문의사항 | 상황 파악 |
| consent | 개인정보 동의 | 개인정보 동의 | 법적/신뢰 요건 |

---

## 6-5. Preparation State

검진 전 준비를 돕기 위한 상태 모델이다.

| 필드 | 예시 | 경험 목적 |
|---|---|---|
| requiresID | true | 신분증 안내 |
| requiresFasting | true/false/contextual | 금식 안내 |
| requiresCompanyForm | true/false | 배치전검진 서류 안내 |
| resultDeliveryMethod | 방문/문자/별도 문의 | 결과 안내 |
| visitTimeNotice | 오전 권장/예약 필요 | 방문 전 혼란 감소 |

---

# 7. 페이지 & 섹션 구조

## 7-1. 필요한 페이지

| 페이지 | 역할 | 주요 사용자 |
|---|---|---|
| Home | 전체 목적 분기와 핵심 정체성 전달 | 전체 |
| 출장검진 | 기업/기관 출장검진 안내 | 기업 담당자 |
| 배치전·배치후검진 | 개인 근로자 검진 안내 | 개인 근로자 |
| 특수건강검진 | 특수검진 대상 및 기업 상담 | 기업/개인 |

| 검사·장비 | CT, 초음파, 내시경 등 신뢰 보강 | 전체 |

| 콘텐츠 상세 | 블로그/가이드 내용을 서비스로 연결 | 검색 유입 사용자 |

| 완료 안내 | 제출 후 다음 단계 안내 | 전환 사용자 |

## 7-2. 전역, 모든 페이지 공통

모든 페이지에 공통으로 적용되는 구조다.

### Global Header

역할:

```txt
현재 위치를 보여주고, 주요 동선으로 빠르게 이동하게 한다.
```

필요 요소:

- 로고
- 주요 메뉴
- 기업검진 문의 버튼
- 개인검진 예약 버튼
- 전화 CTA
- 모바일 햄버거 메뉴

### Floating CTA

역할:

```txt
사용자가 어느 페이지에서든 문의/예약 행동으로 이동할 수 있게 한다.
```

구성:

```txt
전화 문의
카카오 상담
기업검진 문의
개인검진 예약
```

### Breadcrumb

역할:

```txt
사용자가 현재 자신이 어느 정보 단위 안에 있는지 이해하게 한다.
```

### Related Service Block

역할:

```txt
콘텐츠와 서비스 페이지가 단절되지 않게 한다.
```

### Global FAQ

역할:

```txt
검진 전 공통 불안을 줄인다.
```

### Footer

역할:

```txt
병원 신뢰 정보와 기본 접점 제공.
```

포함 요소:

- 병원명
- 주소
- 진료시간
- 전화번호
- 사업자/의료기관 정보
- 개인정보처리방침
- 오시는 길

## 7-4. Content Detail, 콘텐츠 상세

기존 `Product Detail`은 이 프로젝트에서 **콘텐츠 상세 페이지**로 변경한다.  
이 페이지는 블로그/건강정보 글이 단순히 끝나는 곳이 아니라 관련 서비스로 연결되는 접점이다.

### Section 1. 콘텐츠 제목 및 요약

필요 요소:

- 제목
- 핵심 요약
- 카테고리
- 관련 사용자 태그

### Section 2. 핵심 질문 요약

예시:

```txt
이 글에서 해결하는 질문
- 배치전검진은 무엇인가요?
- 준비물은 무엇인가요?
- 결과는 언제 나오나요?
```

### Section 3. 본문

원칙:

- 의료 정보는 명확하게
- 과장 표현은 피함
- 긴 문단보다 짧은 정보 블록 사용

### Section 4. 관련 서비스 연결

예시:

```txt
이 글과 관련된 서비스
→ 배치전·배치후검진 안내
→ 개인 배치전검진 예약
```

### Section 5. 관련 FAQ

목적:

```txt
본문을 읽고도 남는 실무적 질문을 해결한다.
```

### Section 6. 하단 CTA

사용자 유형에 맞게 다르게 제공한다.

```txt

채용검진
국가검진
```

```txt

담당자명

사업장 위치
예상 검진 인원

```

```txt
이름
연락처
검진 목적
방문 희망일
문의 내용
```

### Step 4. 안내 확인

사용자에게 제출 전 안내한다.


완료 화면은 단순 “제출되었습니다”가 아니어야 한다.

필요 요소:

```txt
문의가 접수되었습니다.
담당자가 확인 후 연락드립니다.
방문 전 준비물과 검진 가능 여부는 상담 시 안내됩니다.
```

추가 CTA:

```txt
진료시간 확인
오시는 길 보기
관련 FAQ 보기
```

# 11. 설계원칙, UX 레벨

## Principle 1. 목적을 먼저 묻는다

사용자에게 병원의 모든 정보를 보여주기 전에, 사용자가 왜 왔는지 먼저 분리한다.

```txt
기업 담당자인가?
개인 근로자인가?
지역 진료 목적인가?
```

---

## Principle 2. 기업과 개인의 동선을 섞지 않는다

기업 담당자와 개인 근로자는 같은 검진을 찾더라도 기대하는 정보가 다르다.

```txt
기업 → 일정, 인원, 출장 가능 여부, 협약, 결과 전달
개인 → 당일 가능 여부, 준비물, 금식, 결과 시점, 비용
```

---

## Principle 3. 서비스명보다 사용자의 질문을 우선한다

사용자는 “특수건강검진”이라는 명칭보다 자신의 상황을 먼저 생각한다.

```txt
“입사 전에 검진이 필요합니다.”
“야간근로자 검진 대상입니다.”
“사업장으로 검진을 부르고 싶습니다.”
```

---

## Principle 4. 절차를 보이면 불안이 줄어든다

검진은 의료 행위 자체보다 전후 절차가 불안을 만든다.  
모든 주요 서비스 페이지는 절차를 기본 구조로 제공한다.

---

## Principle 5. 준비물과 결과 시점을 명확히 안내한다

개인 근로자에게 가장 중요한 UX는 “방문 전에 뭘 준비해야 하는지”다.  
따라서 준비물, 금식, 결과 시점은 별도 컴포넌트로 반복 제공한다.

---

## Principle 6. 콘텐츠는 끝점이 아니라 전환의 중간 지점이다

블로그/건강정보 글은 읽고 끝나는 것이 아니라 관련 서비스, FAQ, 문의/예약으로 연결되어야 한다.

---

## Principle 7. 과장보다 명확함이 신뢰를 만든다

의료 웹사이트는 과한 카피, 공포 마케팅, 장비 자랑보다 명확한 조건과 절차가 더 강한 신뢰를 만든다.

---

## Principle 8. 모바일 우선으로 설계한다

배치전검진, 당일검진, 전화 문의 사용자는 모바일에서 빠르게 행동할 가능성이 높다.

필수 요소:

```txt
하단 고정 CTA
전화/카카오 즉시 연결
짧은 카드형 정보
접히는 FAQ
간단한 문의 폼
```

---

## Principle 9. 완료 후에도 경험을 닫지 않는다

문의 완료 후 사용자는 아직 불안하다.  
완료 화면은 다음 단계를 알려주는 안정감의 마지막 접점이어야 한다.

```txt
접수 완료
→ 담당자 확인
→ 연락 예정
→ 방문 전 준비사항
→ 진료시간/오시는 길
```

---

## Principle 10. 병원은 장소이면서 동시에 현장으로 이동하는 시스템이다

스마트허브병원의 핵심은 정적인 병원 소개가 아니라, 사업장과 현장을 연결하는 검진 운영 능력이다.

따라서 UX는 다음 정체성을 유지해야 한다.

```txt
오세요
```

이 아니라,

```txt
필요한 검진을 병원과 현장 어디에서든 연결합니다
```

---

# 12. 최종 UX 정의

스마트허브병원 웹사이트의 UX는 사용자가 의료 정보를 많이 읽게 만드는 것이 아니라,  
자신의 검진 상황을 빠르게 이해하고, 필요한 준비를 확인하고, 적절한 문의 또는 예약으로 안정적으로 이동하게 만드는 경험이다.

최종 정의는 다음과 같다.

```txt
스마트허브병원 UX는
기업 담당자와 개인 근로자의 검진 문제를 분리해 이해시키고,
절차와 준비사항을 명확히 보여주며,
문의와 예약 이후까지 안정적으로 안내하는
산업·현장 건강검진 경험 설계이다.
```
